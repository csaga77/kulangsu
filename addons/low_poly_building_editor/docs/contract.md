## Current contract

- The building editor is a Godot editor plugin and must stay out of the runtime app shell.
- `Building3D` is the scene-owned root for one building assembly. A scene may contain multiple independent roots or packed building scene instances; selecting a root or any authored descendant makes it the active target, and the dock's Add Building command always creates and selects a new uniquely named root. Each root owns only its direct building-part lookup, geometry-change detection, and transient geometry resolvers, so merging and clipping never cross into a sibling building. No building-editor configuration is exported on the root; tool grid values, drawing constraints, and new-element defaults remain editor-dock state.
- `BuildingFactory` is the stateless construction boundary for wall, room, floor, stairs, standard rail, pillar, roof, and typed door/window nodes. It owns the concrete pillar/roof/opening style registries, style-property application, unique child naming, shared placement snapping/constraining helpers, and segment-aware opening transforms. Callers pass a `Building3D` or wall naming context and explicit values; the factory returns a configured node without parenting it or editing undo history. Both editor commits and generated buildings use this boundary. `create_stairs_node()` accepts optional trailing left/right rail enable flags, the same standard rail dimension values `create_rail_node()` accepts, and a trailing `rail_edge_margin`, defaulting every rail argument to disabled/standard values so existing callers are unaffected.
- `BuildingSpec` is the versioned typed input contract for deterministic batch generation. JSON is an interchange format for agents and scripts; parsing produces a typed resource and validates schema/generator versions, dimensions, style categories, opening height bounds, and version-specific scope before compilation.
- `BuildingSpecCompiler` consumes a valid `BuildingSpec`, resolves every `random` style and footprint jitter from the supplied seed, creates one detached `Building3D` assembly through `BuildingFactory`, assigns authored scene ownership, validates openings through `Building3D.can_place_wall_opening()`, refreshes transient geometry clips, and returns the complete building plus resolved values, errors, and warnings. Compilation failures return no partial building.
- `generate_building.gd` is the headless JSON-to-`.tscn` entry point. It stays an authoring tool outside the runtime app shell, prints a machine-readable report, optionally writes the report, creates output directories, and saves only successful compiler results.
- `BuildingThumbnailRenderer` owns the reusable isolated `SubViewport`, entrance-aware isometric camera, preview ground, lighting/environment, two-frame renderer warm-up, background-only image rejection, PNG output, and contact-sheet composition used by visual generation. It never changes the packed building scene.
- `generate_variants.gd` duplicates one valid base spec across consecutive seeds, compiles and saves each successful building, asks `BuildingThumbnailRenderer` for one fixed thumbnail per variant, and writes a versioned manifest mapping indices/seeds/resolved parameters to scene and image paths. Its contact sheet uses successful manifest order, and per-variant failures remain explicit rather than being silently omitted.
- Visual variant generation requires a graphical display/rendering driver. It fails before generation when Godot uses the dummy headless display; automation may use a tiny off-screen native window. Scene-only `generate_building.gd` remains fully headless.
- Generator schema/generator version 1 supports one rectangular storey, one required exterior entrance, repeated facade windows, optional porch pillars, footprint-cell jitter, and one roof. It intentionally does not infer internal rooms, upper-storey circulation, props, compound wings, or overlapping roof plans.
- `WallGeometryResolver` owns collinear wall merge detection, intersecting-wall detection, opening-placement conflict checks, geometry-only sibling clip assignment, and propagation of openings across collinear sibling walls. `RoofGeometryResolver` owns same-eave roof overlap calculations, visible-area tests, covered-region refresh, polygon clipping, and transient roof underside surfaces supplied to walls. Resolver instances are implementation services created from one building root's current children; they are not nodes, serialized authored state, or independent building owners.
- Selecting authored `Wall3D`, `Floor3D`, `Stairs3D`, `Rail3D`, `Pillar3D`, `Roof3D`, any `BuildingOpening3D` subtype, or a prop placed by the plugin updates the active building tool mode through the dock/toolbar path; typed `Window3D` and `Door3D` branches determine the opening mode, legacy metadata identifies base-class openings, and placed prop roots carry a persistent plugin marker.
- Debug wireframe is a dock-owned global display preference, not authored block state or a factory argument. `BuildingWireframe3D` discovers each root or multipart source `MeshInstance3D` and temporarily assigns a native spatial-shader `wireframe` pass directly to its RenderingServer instance, bypassing authored node properties and scene mutation. It must not add debug nodes to the edited scene tree, extract triangle faces, deduplicate edges on the CPU, or allocate duplicate mesh geometry. Existing authored material overlays are chained as the debug material's next pass and restored when debug display is disabled. Normal wireframes retain back-face culling and depth testing so invisible surface triangles remain hidden; X-ray mode explicitly disables only depth testing and is the opt-in path for hidden geometry. `BuildingMesh3D` provides the shared transient display API for all present and future mesh blocks, `BuildingOpening3D` applies the same API to its generated child parts, and marked prop roots use the recursive utility directly. Changing enabled state, color, or X-ray mode must not rebuild source mesh, generated parts, collision, authored properties, or scene hierarchy.
- The dock owns shared creation defaults and selected-style defaults. It shows only controls applicable to the active tool and selected concrete style, persists those values in editor project metadata, and applies them consistently to previews and committed nodes without serializing defaults on `Building3D`.
- `Wall3D` stores zero or more typed `WallSegment3D` resources in its exported `segments` array. Segment zero, when present, supplies the node transform; every segment stores its own parent-local endpoints, height, thickness, and color. Changing the node-level wall height, thickness, or color applies that value to every authored segment, while editing a segment resource directly can still create intentional per-segment variation. An empty array is valid and generates no mesh or collision. Hidden `start_point`, `end_point`, and `extra_segments` compatibility aliases load legacy scenes but are not inspector-facing authored state. Shared joint endpoint drags move every connected segment endpoint together, single endpoint drags can disconnect from or reconnect to a shared joint, zero-length endpoint drags can promote another segment to index zero when the old first segment is deleted, and one low-poly mesh/collision assembly rebuilds from authored segments, sibling clip spans, roof clip surfaces, and direct child openings.
- Wall meshes miter shared non-collinear endpoints by extending wall side faces to their endpoint-direction side-line intersections without adding butt-style joint caps, keep those miter points invariant when the same span is drawn in the opposite start/end direction, keep top caps tied to wall footprints with targeted overlap clipping so 3+ split joins render filled while enclosed wall loops keep their room interiors open, omit hidden underside triangles on the shared wall base plane, and clip overlapping faces in plan space so crossings, T-junctions, and corners avoid square butt seams, open miter gaps, and z-fighting caps. Down-facing header surfaces above openings remain because they can be visible. Where a visible roof intersects a wall, geometry above the roof bottom is removed while the wall volume and side faces below it are retained across eaves, slopes, and gable ridges; when the roof covers only a middle interval, the uncovered wall spans on both sides remain complete and full-height. The covered clipping boundary intentionally generates no wall upper-face triangles because the roof supplies that surface. Retained exterior wall faces keep their original horizontal outward normals and matching winding.
- The dock persists a Wall Type of `Wall` or `Room` plus a Room-only Sides value clamped to a minimum of three. Wall mode draws one grid-snapped span and retains optional eight-way direction locking. Room mode treats the grid-snapped gesture points as opposite polygon bounds and creates one `Wall3D` node with the configured number of connected authored spans, one generated mesh/collision assembly, and one undo/redo action. Four sides preserves the axis-aligned rectangular room contract; other side counts distribute vertices around the bounded ellipse.
- When separate rooms or walls contain matching collinear overlapping spans, scene order always owns the overlap: the earlier wall keeps the shared mesh/collision and the later wall clips its duplicate portion. Authored spans remain unchanged so moving either room restores the appropriate geometry.
- Door and window placement is allowed on either wall of a matching collinear shared room boundary, independently of scene order; the wall that renders the shared span cuts the opening (authored or propagated from the clipped sibling). Crossings, T-junctions, and different-thickness spans reject the opening.
- An opening node parented to any wall of a matching collinear overlap is cut from every collinear sibling that shares the span, in both scene-order directions: `WallGeometryResolver` maps the opening into each receiving wall's segment-local span. The scene-earlier wall renders the shared mesh and cuts the visible hole; because collision is generated per wall and is not sibling-clipped, each collinear wall also cuts the opening from its own collision so the duplicated overlap collision does not block the doorway. The mapping is transient rebuild data passed alongside the sibling clip segments, never edits authored spans, and never reparents the opening node; opening size and local-transform signals refresh the affected walls.
- Wall drag edits are segment-aware: Shift-clicking a wall body inserts a joint by splitting that span into two connected segments, dragging an isolated endpoint changes only the picked segment endpoint, dragging a shared joint endpoint changes every connected segment endpoint at that joint, Option/Alt-dragging a shared joint endpoint disconnects only the picked endpoint, dragging a single endpoint near another endpoint or joint snaps it there to connect, dragging a wall body translates the wall's authored segment set, zero-length endpoint drags delete the collapsed segment or wall node through undo/redo, drag commits normalize same-wall intersections, and crossings with other wall instances refresh sibling clip geometry without merging wall nodes.
- Dragging the body of one side of a four-span rectangular room resizes that room perpendicular to the selected side, moves both connected corner endpoints, leaves the opposite side fixed, preserves opening anchors, rejects a zero-width or zero-depth result, and commits through one undo/redo action. Option/Alt-dragging a rectangular room body bypasses side resizing and translates the full authored segment set instead, grid-snapped, preserving opening anchors, through one undo/redo action. Body dragging an irregular or open wall network continues to translate the full authored segment set.
- `Floor3D` supports rectangle-corner storage and general polygon-point storage without a serialized `floor_style` property. Empty `polygon_points` selects `start_point/end_point` rectangle storage; any authored polygon points select polygon storage. Rectangle and Polygon in the dock are creation modes shared by Solid and Hole: Rectangle uses the two-corner or drag gesture, while Polygon grid-snaps each click and closes when the first vertex is clicked or Enter is pressed. Solid footprints and authored polygon-hole outlines expose the same vertex/edge/body editing model. Dragging one vertex moves only that vertex, dragging an edge moves its two adjacent vertices together, Shift-clicking an edge inserts a snapped vertex, Option/Alt-clicking a vertex removes it while preserving the three-vertex minimum, and body dragging translates the full outline. A rectangle-corner floor without holes converts to polygon-point storage on its first shape edit; undo restores its rectangle storage. New holes serialize in `floor_hole_polygons`; legacy rectangular `floor_holes` remain supported for existing scenes. Both outer storage forms derive the node transform from their minimum X/Z top corner, extend thickness downward, and rebuild one low-poly slab mesh plus generated concave collision around rectangle or polygon holes. Holes must remain fully inside the floor and may not overlap another hole. Invalid/self-intersecting polygons and zero-area outlines are rejected, and every creation, move, reshape, insertion, and removal commits through undo/redo.
- `Stairs3D` stores opposite parent-local footprint corners in `start_point/end_point`, derives its transform from the minimum X/Z lower-entry corner plus `stair_rotation_degrees`, and rebuilds one stepped low-poly mesh plus generated concave collision from total height, step count, underside thickness, rotation, and color. The broad hidden underside quad and its two triangles are omitted; the front/back boundaries and side profiles still extend to the configured underside depth. Generated side-wall box collision blockers sit inside the left and right footprint edge strips under the treads and are split per tread so their top edges follow the stair profile; `HumanBody3D` cannot enter the stairs from the side, but a character already on a step can leave or jump off sideways without hitting a full-height invisible rail or collision outside the stair footprint. Stairs tool creation converts the first draw point and current draw point into opposite corners of the rotated footprint. Stairs tool edits move the body, resize edges/corners on the active stairs grid in the stairs' rotated local frame, rotate around the footprint center, preserve the existing lower-entry Y, reject zero-area footprints, and commit through undo/redo. `Stairs3D` additionally stores independent `left_rail_enabled`/`right_rail_enabled` flags, a `rail_edge_margin` inset, plus standard post-and-bar rail dimensions (height, post thickness, rail thickness, lower-rail center height, color) matching `Rail3D`'s property names (it has no post-spacing property); when enabled, a side rail is appended into the same generated stairs mesh and concave collision using the shared `StandardRailGeometry` strategy, following a straight raked line from the lower entry to the top tread rather than stepping with each tread, with one flat, upright post centered on each tread's depth span and based on that tread's actual surface height instead of an evenly-spaced distribution following the raked line's diagonal. `rail_edge_margin` insets each enabled rail's origin from the footprint's left/right edge instead of straddling it, clamped to at most 45% of the footprint width so opposing margins on a narrow stairs cannot cross.
- `Rail3D` stores one parent-local `start_point/end_point` span and standard post-and-bar properties: total height, maximum post spacing, square post thickness, square rail thickness, lower-rail center height, color, rebuild state, and collision behavior. Its transform follows the horizontal span from the authored start point. Both endpoints always receive posts; intermediate posts are distributed evenly so their spacing never exceeds the configured maximum. The top handrail and optional lower rail extend across the span. Rail tool creation and edits snap to the active Rail grid, preserve the base Y, support body translation and endpoint resizing, reject undersized spans, and commit through undo/redo. `Rail3D` and `Stairs3D`'s optional side rails both delegate bar/post box assembly to `standard_rail_geometry_3d.gd`'s stateless `StandardRailGeometry.append_rail()`, which embeds a canonical level post-and-bar profile into the caller's run/up/side axes and applies a linear height shear for a nonzero rise, so a level standard rail (rise `0.0`) and a straight raked stairs rail (rise equal to the stair height) share one geometry implementation instead of duplicating it per caller. An optional explicit `post_positions` argument overrides the default spacing-based distribution: `Rail3D` omits it (even spacing, both endpoints always posted), while `Stairs3D` supplies one position per tread from `StandardRailGeometry.tread_mid_post_positions()`, centering each post on that tread's depth span. Posts remain upright with flat bases regardless of rise, but each top edge follows the sloped handrail underside across its footprint. The handrail underside is partitioned around matching post openings, so the two parts share an exact geometric boundary without overlapping internal faces or a wedge gap. The post's enclosed top cap and hidden bottom cap are both omitted, removing four triangles per post while retaining the welded side-face top boundary and bottom contact edges. An optional parallel `post_base_heights` argument gives each post's flat bottom an explicit absolute height instead of the bar/lower-rail's rise/length diagonal. `Rail3D` omits it (posts sit on the level ground plane). `Stairs3D` supplies `StandardRailGeometry.tread_mid_post_base_heights()`, so each post's base rests on its own tread's real top surface instead of the diagonal projection at its mid-tread run position (which would otherwise be half a riser too low, embedding the post partway into the step).
- `Pillar3D` stores only universal pillar state: parent-local `base_point`, lower radius, optional upper-radius override, height, upper/lower rim settings, color, rebuild state, and collision behavior. `RoundPillar3D` and `TaperedPillar3D` inherit the internal variable-sided layer that owns `side_count`; `SquarePillar3D` and `OctagonalPillar3D` own their fixed geometry without exposing that unused control. Pillar tool edits move the body or resize radius on the active pillar grid, preserve the existing base Y, reject undersized radii, and commit through undo/redo.
- `Roof3D` stores universal roof state: opposite parent-local rectangle corners, thickness, overhang, rotation, color, covered render regions, rebuild state, and collision behavior. `FlatRoof3D`, `ShedRoof3D`, `GableRoof3D`, `HipRoof3D`, and `DomeRoof3D` identify the serialized style; only `FlatRoof3D` additionally owns optional parent-local `polygon_points`. Empty Flat points use rectangle-corner storage, while non-empty points use polygon storage. Rectangle and Polygon in the Flat-roof dock controls differ only during creation. Both expose the same outline editing: drag a vertex, drag an edge and its two endpoints, Shift-click an edge to insert a snapped vertex, Option/Alt-click a vertex to remove it, or drag the body to translate the footprint. The first outline reshape of a rectangular Flat roof promotes its four corners to polygon storage, and undo restores the rectangle representation. Shed, gable, hip, and dome roofs remain rectangular. The internal sloped-roof layer owns the face-angle property used by those styles; only `HipRoof3D` owns the hip gable drop. Each style delegates its generated height, surface sampling, top faces, topology, and edge splits to its matching geometry strategy. Floor slabs and unclipped Flat polygon roofs share `PolygonPrismGeometry` for winding normalization, triangulation, top/underside faces, direct boundary extrusion, and collision faces. Flat overhang computes one bounded miter point per authored point instead of accepting a clipping-library bevel or split contour, so acute footprints retain one render corner per authored corner. Covered regions may still introduce transient mesh intersection points without changing authored polygon storage. Dome roofs use a deterministic elliptical cap made from 16 radial facets and five vertical bands; the rectangular footprint and overhang define its ellipse, the configured angle defines its peak rise from the shorter radius, and a 45-degree square dome is hemispherical. Simple hip roofs use a horizontal ridge along the longer footprint axis, two triangular short faces, two trapezoid long faces, and the same configured angle on all faces; square hip footprints degenerate to a centered apex. Positive hip gable drop clips the hip ridge ends downward from the peak and extends the ridge as needed to preserve the configured face angle, producing a half-hip or jerkinhead variant. Roof tool edits preserve the existing eave-plane Y, reject invalid or zero-area footprints, recompute sibling roof covers and wall clipping, and commit through undo/redo. Rectangular and pitched roofs retain rotated edge/corner resize and center rotation behavior; polygon Flat roofs derive orientation from their authored vertices.
- `BuildingOpening3D` is the shared persistent child-node base for window and door holes. Its `opening_width`, `opening_height`, segment-local position, frame properties, and collision settings define the common opening rectangle and casing contract consumed by the wall mesh compiler. Openings are mapped to the nearest wall segment and restored into that segment's updated frame when wall geometry changes (including endpoint, joint, extra-segment, and whole-wall drags).
- `Window3D` owns only behavior applicable to every window opening and generates no style geometry. The internal `window_pane_3d.gd` layer owns pane depth/color plus plain-pane primitives for glazed pane-bearing styles. Every concrete class owns its geometry entry point: single/double windows choose their pane layout, `GridWindow3D` builds its muntins, `LouveredWindow3D` builds its slats, `TransomWindow3D` builds its rail, `ArchedWindow3D` builds its stepped fillers, and `WindowFrame3D` intentionally builds no pane content.
- `Door3D` owns only behavior and defaults applicable to every door opening and generates no style geometry. The internal `door_leaf_3d.gd` layer owns panel depth/color plus plain-leaf primitives for styles with physical leaves. Every concrete class owns its geometry entry point: single/double doors choose their leaf layout, `GlazedDoor3D` builds its solid/glass split, `GlazedGridDoor3D` adds its lite muntins, `PanelDoor3D` builds its inset grid, `DutchDoor3D` builds its split leaves and mid rail, and frame styles intentionally build no leaf content.
- The editor must instantiate the concrete class for the selected style for both previews and committed nodes; style properties and style geometry remain in that class, not in conditionals on `BuildingOpening3D`, `Window3D`, or `Door3D`. Primary parts (panes, panels/leaves, slats) carry collision while decorative overlays do not, and style details never change the opening rectangle the wall compiler reads. `frame_sides`, `frame_protrusion`, and `wall_thickness` remain shared base properties and only change the casing depth.
- Legacy scenes that stored solid `door_panel_count`, `door_panel_depth`, or `door_panel_color` values directly on `BuildingOpening3D` continue to deserialize those values through storage-only compatibility properties. New authoring must use a `Door3D` subtype.
- New overlapping collinear walls of matching thickness and height merge by extending the retained wall endpoints instead of creating duplicate spans.
- Non-collinear intersecting walls sharing a base plane remain separate authored `Wall3D` nodes: sibling spans are supplied as non-rendered clip segments, each wall renders only its own authored spans, buried side and top-cap geometry is removed in plan space, same-height top-cap ownership follows scene order, roof underside cuts are supplied as transient non-rendered clip surfaces, and undo/redo refreshes clip geometry without reparenting children or removing other wall nodes. Heights from sibling walls may split a wall's generated cells only when their plan footprints overlap; unrelated walls must not introduce mismatched roof-clip contours.
- `WallSegment3D`, `Wall3D`, `Roof3D`, and `BuildingOpening3D` emit authored-geometry signals for clipping-relevant changes. `Building3D` dynamically connects direct wall/roof children, coalesces those notifications into one deferred update, and asks the transient resolvers to refresh clipping data without per-frame signature polling. Resolver-derived clip assignments and covered regions stay silent so they cannot create refresh feedback loops.
- Serialized procedural geometry is a validated cache, not authored state. `BuildingMesh3D` supplies hidden source/clipping signatures and cache helpers shared by `Wall3D`, `Floor3D`, `Stairs3D`, `Rail3D`, `Pillar3D`, and `Roof3D`; scene loading reuses a matching root mesh and recreates generated collision/debug children independently. Generated mesh and material resources plus authored `WallSegment3D` resources are scene-local, so editing one packed `Building3D` instance cannot alter another instance's geometry or cache. Wall sibling/roof clips and roof covered regions participate in clipping signatures, so unchanged coordinator refreshes do not rebuild meshes while changed clipping conditions do. `BuildingOpening3D` stores an equivalent signature plus compact generated part-mesh descriptors because its window/door styles consist of multiple child meshes rather than one root mesh; load recreates child nodes and collision from those cached resources without rerunning style geometry. Legacy scenes without cache metadata rebuild once.
- Editor commands that create, move, resize, restore, undo, or redo walls and roofs refresh the complete building clipping chain synchronously: roof covered regions resolve first, then wall sibling and roof-underside clips. Partial wall-only or roof-only command refreshes must not rely on a later signal turn to repair cross-family clipping.
- Editor integration is event-driven: tool-mode changes push active state into viewport overlays, overlay parent-order notifications keep active overlays in front, and native viewport button signals maintain mutual exclusivity with building tools. `Wall3D` uses restartable one-shot editor rebuild requests to update visuals immediately and defer collision rebuilding without an idle `_process()` loop.
- Generated wall solid collision cells, exact concave collision from roof-clipped retained wall faces, floor/stairs/rail/pillar/roof collision, floor-hole side geometry, and opening frame parts are rebuild artifacts identified by metadata; they are not author-facing source nodes.

## Building Block Style Pattern

Use this pattern for every future serialized building-block family that has distinct styles:

1. The family base class owns only properties and behavior applicable to every style. It may provide low-level mesh, material, collision, lifecycle, and transform primitives, but it must not export a style selector or branch on style to generate geometry.
2. Add an internal intermediate class only when a real subset shares authored properties or primitives. Examples are pane-bearing windows, leaf-bearing doors, variable-sided pillars, and sloped roofs. Styles outside that subset must not inherit unused inspector properties.
3. Every serialized style has a concrete class. That class owns style identity, style-only exported properties, defaults, and its geometry entry point. Large reusable shape math may live in a one-to-one style geometry strategy, but it must remain selected by the concrete class.
4. Editor factories map persisted dock style keys to concrete scripts. Both previews and committed nodes come from the same factory; changing a preview style recreates the preview with the correct class instead of mutating a style flag.
5. Compatibility APIs may route an old style key to the concrete strategy for legacy consumers, but routing and storage adapters must be isolated from the new-authoring object model. They must not reintroduce style properties or geometry branches into the family base.
6. Frame-only or empty styles intentionally override the geometry entry point with no inner content. Shared dimensions consumed by another compiler—such as an opening rectangle read by `Wall3D`—remain on the family base and cannot be changed by decorative style geometry.
7. Tests must verify inheritance, inspector-property ownership, factory class selection, each style's generated geometry, and compatibility behavior. A base-class property regression is a contract failure even when the generated mesh still looks correct.
8. A tool option that performs an operation is not automatically a serialized style. `Room` creates a four-segment `Wall3D`, and `Floor Hole` modifies a `Floor3D`; neither needs a `Room3D` or `FloorHole3D` subtype unless it becomes persistent polymorphic scene state.

Before adding a style switch to an existing block, first decide whether the distinction persists in the scene. Persistent polymorphism follows this hierarchy; one-shot editor operations remain coordinator/plugin workflows.

## Governance

- Keep building-editor scene mutations undoable through `EditorUndoRedoManager`; placed wall, room, floor, stairs, rail, pillar, roof, prop, and opening mutations — including deletion of a zero-length wall segment or its final wall node — must go through it so undo/redo restores scene hierarchy and rebuilt meshes together.
- If wall segment storage, floor style/vertex/corner/hole storage, stairs footprint storage, rail span/component storage, pillar base/radius storage, roof footprint storage, segment splitting, child reparenting, opening semantics, generated collision, merge behavior, or editor interaction changes, update this contract and [`feature.md`](feature.md).
- If the building-spec or variant-manifest schema, deterministic resolution order, supported generator scope, report fields, preview camera/lighting contract, or save-on-success behavior changes, increment the appropriate schema/generator version and update this contract, [`feature.md`](feature.md), the example spec, and generator regression coverage together.
- Do not treat `Building3D` buildings as runtime story targets or save/resume anchors without updating the low-poly 3D prototype contract in [`../../../docs/contracts.md`](../../../docs/contracts.md) and the relevant gameplay docs.
