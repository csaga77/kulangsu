# Low-Poly Building Editor

## Goal

- Provide a Godot-native editor plugin for quickly authoring grid-snapped low-poly building blockouts.
- Keep building authoring in normal scene trees so generated walls, openings, props, and transforms serialize into `.tscn` files.
- Provide a deterministic, versioned JSON-to-`.tscn` path so agents and batch tools can author buildings through the same typed construction API as the editor.
- Produce consistent visual variant sheets so agents and authors can compare seeded outputs before selecting or mutating a building spec.

## User / Player Experience

- This is a developer-facing tool, not runtime player UI.
- Agents can provide a `BuildingSpec` JSON file and run the headless generator to produce a normal editable `Building3D` scene plus a machine-readable resolution/validation report.
- Agents can run the graphical variant command to receive consecutive seeded scenes, entrance-aware isometric thumbnails, a versioned manifest, and a contact sheet in matching order.
- Authors can run the variant gallery test scene to inspect 12 generated buildings together and step through adjacent deterministic seed batches without first writing generated scenes to disk.
- Authors enable the `Low-Poly Building Editor` plugin, then choose `Wall`, `Floor`, `Stairs`, `Rail`, `Pillar`, `Roof`, `Door`, `Window`, or `Prop` from either the dock's tool dropdown or the building tool buttons added to the 3D viewport toolbar, and place content directly in the 3D viewport.
- The 3D viewport toolbar exposes one icon-only toggle button per building tool (next to Godot's native Transform/Move/Rotate/Scale/Select buttons, with tool names in hover tooltips) so authors can switch tools without moving focus to the dock; the toolbar and dock stay in sync, so selecting a tool in either place updates both.
- Selecting an authored `Wall3D`, `Floor3D`, `Stairs3D`, `Rail3D`, `Pillar3D`, `Roof3D`, or `BuildingOpening3D` subtype in the editor automatically switches the dock and 3D viewport toolbar to that block's editing mode. `Window3D` and `Door3D` inheritance identifies the opening mode directly.
- The building tool buttons are mutually exclusive with the native 3D viewport selection modes (Transform, Move, Rotate, Scale, Select): activating a building tool clears the native mode highlight, and clicking any native mode button (or its shortcut) deactivates the building tool and returns to that native mode. The native `Select` mode is the resting "no building tool" state, so the toolbar does not duplicate a Select button.
- Hovering over the selected tool name in the dock opens a tooltip containing that tool's shortcuts. The dock body shows live status text, the selected tool's shared creation defaults, and a `Style Properties` subsection containing only properties owned by the selected pillar, roof, window, or door style.
- Wall drawing supports click-start/click-end and click-drag-release flows with live snapped preview geometry. The persisted Wall Type chooses one wall span or an enclosed room; Room also persists a Sides value with a minimum of three and a default of four. New walls and rooms draw on the dock's persisted parent-local base Y plane.
- Floor drawing has a persisted creation Style of Rectangle or Polygon shared by both Floor Types. Rectangle supports click-corner/click-corner and click-drag-release flows; Polygon captures grid-snapped vertices until the first vertex is clicked again or Enter is pressed. Both show live preview geometry and share one editing model after creation. Floor Type chooses a solid slab or a matching rectangle/polygon hole cut into an existing floor.
- Stairs drawing supports click-corner/click-corner and click-drag-release flows with live snapped preview geometry; new stairs are rectangular stepped solids whose lower entry sits on the dock's persisted parent-local base Y plane and whose top tread reaches the configured total height.
- Rail drawing supports click-start/click-end and click-drag-release flows with live snapped preview geometry; new standard rails contain square end/intermediate posts, a top handrail, and a lower rail on the dock's persisted parent-local base Y plane.
- Pillar placement supports a click-to-place workflow with live snapped preview geometry; new pillars use the selected low-poly style, optional upper/lower rim settings, and their base sits on the dock's persisted parent-local base Y plane.
- Flat Roof drawing has a persisted Rectangle or Polygon footprint creation choice. Rectangle supports click-corner/click-corner and click-drag-release; Polygon captures grid-snapped vertices until the first vertex is clicked again or Enter is pressed. Other roof styles keep rectangle creation. All roof eaves sit on the dock's persisted parent-local base Y plane.
- Prop, door, and window placement uses translucent green/red previews before committing nodes.
- While the cursor is over a 3D viewport, `R` rotates the prop preview in 90-degree steps (prop mode only) and `Escape` or right-click cancels the active preview.
- The prop palette folder is configurable in the dock, defaults to `res://assets`, and includes `.tscn`, `.scn`, `.gltf`, and `.glb` scene assets. The configured folder persists across editor sessions via editor project metadata.

## Rules

- `Building3D` is the root node for one building assembly; scenes may contain multiple independent roots or packed building instances, and clipping/merging stays inside each root.
- `BuildingFactory` is the shared detached-node construction boundary for manual editor commits and generated buildings, including typed opening styles and segment-aware opening transforms.
- Generator version 1 accepts one rectangular storey, seeded footprint jitter and `random` style values, one required entrance, repeated validated facade windows, optional porch pillars, and one roof. Unsupported versions, style categories, dimensions, and openings fail before a scene is saved.
- `BuildingSpecCompiler` assigns authored ownership, uses `Building3D.can_place_wall_opening()` for every door/window, refreshes final clipping, and returns no partial building after a compilation error.
- Visual batches use one isolated preview world and consistent lighting while framing each thumbnail from the generated entrance side. Rendering warms up for two frames and rejects background-only images so the manifest cannot mark a blank preview successful.
- The scene-only command works with Godot's dummy headless renderer. The visual batch command requires a graphical renderer and fails before generation in headless display mode.
- `Wall3D` stores zero or more parent-local `WallSegment3D` resources in `segments`; segment zero supplies the node transform, node-level height/thickness/color changes apply to every segment, and an empty array produces no wall mesh or collision.
- `Floor3D` stores either opposite parent-local rectangle corners or simple coplanar polygon vertices. It has no `floor_style` property: empty `polygon_points` means rectangle-corner storage and non-empty points mean polygon storage. New holes use editable local-space `floor_hole_polygons`; legacy rectangular `floor_holes` still load. The node transform uses the footprint's minimum X/Z corner, and the slab mesh extends downward from the configured top surface.
- `Stairs3D` stores opposite parent-local footprint corners in `start_point` and `end_point`; its node transform is rebuilt from the minimum X/Z lower-entry corner plus `stair_rotation_degrees`, and its mesh is rebuilt from total height, step count, underside thickness, and color. It also stores optional `left_rail_enabled`/`right_rail_enabled` flags, a `rail_edge_margin` inset, standard-rail dimensions (height, post thickness, rail thickness, lower-rail height, color, with no post-spacing property), and independent lower/upper newel enable and `Tread`/`Floor` placement settings plus newel thickness. Enabled side rails are appended into the same generated mesh and collision, inset from the footprint's left/right edge by the margin (clamped to at most 45% of the footprint width) instead of straddling it, with one flat, upright regular post centered on each tread's depth span, resting on that tread's actual surface height. Tread-mounted newels replace the corresponding terminal tread post; floor-mounted newels retain it and continue the same post cadence one full tread interval beyond the stair run. Horizontal handrail end sections cover flat newel tops, meet the raked section at the newel edge or stair endpoint, and terminate flush with each newel's outer face.
- `Rail3D` stores parent-local span endpoints and standard-rail dimensions; its transform follows the span, and its mesh/collision rebuild from height, maximum post spacing, post thickness, rail thickness, lower-rail height, and color. Its post/bar mesh assembly comes from the shared `standard_rail_geometry_3d.gd` strategy, which `Stairs3D`'s optional side rails reuse for a straight raked rail along the stair run instead of duplicating the box/post generation code; `Stairs3D` overrides the shared strategy's default spacing-based post distribution with one explicit post position per tread.
- `Pillar3D` stores universal pillar state and mesh/collision infrastructure. Concrete round, square, octagonal, and tapered classes identify each serialized style; only the variable-sided round/tapered branch exposes `side_count`.
- `Roof3D` stores universal rectangle footprint, thickness, overhang, rotation, color, cover, mesh, and collision state. `FlatRoof3D` alone adds optional polygon-point storage; concrete flat, shed, gable, hip, and dome classes identify each serialized style, the sloped branch owns face angle, and only the hip class owns hip gable drop.
- Wall drawing snaps the gesture points to the dock's Wall grid and preserves the configured base Y height for new endpoints. Wall Type `Wall` creates one span and can lock its direction to 45-degree increments. Wall Type `Room` uses the gesture points as opposite bounds of an enclosed polygon and creates one `Wall3D` with the configured number of connected spans. Sides is clamped to at least three; four sides preserves the axis-aligned rectangular room, while other counts distribute vertices around the bounded ellipse.
- Solid Floor drawing snaps to the dock's Floor grid, preserves the configured base Y height, and creates one persistent rectangle or polygon floor node. Hole Floor uses the same creation style on the hit floor's top-surface Y and appends one editable polygon outline through undo/redo. Polygon footprints and holes may be concave but must be simple, non-zero-area, fully contained, and non-overlapping.
- In Floor tool mode, Rectangle and Polygon differ only in creation. Solid floor outlines and hole outlines share the same gestures: dragging one vertex moves only that vertex, dragging an edge moves its two adjacent vertices together, Shift-clicking an edge inserts a snapped vertex, Option/Alt-clicking a vertex removes it unless only three remain, and body dragging translates the full outline. Editing a rectangle floor this way promotes its four corners to polygon storage; rectangle hole creation writes four polygon-hole vertices immediately. Solid mode edits the outer outline, while Hole mode edits hole outlines. Floors containing holes allow body translation but lock outer shape edits so holes stay anchored. Hover highlights body moves blue and vertex/edge edits yellow. Edits snap to the active Floor grid, preserve top-surface Y, reject invalid/self-intersecting/zero-area or out-of-bounds outlines, and commit through undo/redo.
- Stairs drawing snaps to the dock's Stairs grid, preserves the configured base Y height as the lower stair entry, and creates one persistent stairs node per committed rectangle. The dock's Rotation value sets the starting Y rotation for new stairs; while drawing, `R` rotates the preview by 90 degrees and `Shift+R` rotates the opposite direction. In Stairs tool mode, hovering over placed stairs highlights them blue for body moves and yellow for edge/corner resizes. Pressing `R` or `Shift+R` while hovering or selecting placed stairs rotates them around the footprint center through undo/redo. Dragging the body translates the stairs; dragging an edge resizes one footprint axis in the stairs' rotated local frame; dragging a corner resizes both axes. Edits snap to the active Stairs grid, preserve the existing lower-entry Y, reject zero-area footprints, and commit through undo/redo. The Stairs Defaults section adds optional Left Rail and Right Rail checkboxes, independent Lower Newel and Upper Newel toggles with Tread/Floor position selectors, Newel Size, and a stairs-only Rail Margin control. Newly drawn stairs generate the selected straight raked rails/newels using shared Standard Rail Defaults; newel size is clamped to handrail width to keep the welded connection covered. Post Spacing does not apply to stairs rails: regular posts are always placed one per tread, centered on that tread's depth span. Rail Margin insets each enabled rail from the stairs' left/right footprint edge instead of straddling it, and is clamped to at most 45% of the footprint width so opposing margins on a narrow stairs cannot cross.
- Rail drawing snaps both endpoints to the dock's Rail grid and preserves the configured base Y. Post spacing is a maximum: posts are distributed evenly so both endpoints always receive a post. In Rail tool mode, body drags translate the full span and endpoint drags resize it. Hover highlights body moves blue and endpoint edits yellow; edits preserve base Y, reject undersized spans, and commit through undo/redo.
- Pillar placement snaps to the dock's Pillar grid, preserves the configured base Y height, and creates one persistent pillar node per committed click. Built-in pillar styles are `Round`, `Square`, `Octagonal`, and `Tapered`; square and octagonal styles force their side counts, while round and tapered use the configured side count. The lower radius controls the base body radius, and the upper radius controls the top body radius when nonzero; `0` keeps the selected style's default top radius. Upper and lower rim controls set each rim band's height and radius outset; setting either rim value to `0` disables that rim. In Pillar tool mode, hovering over a placed pillar highlights it blue for body moves and yellow for radius resizes. Dragging the body translates the pillar; dragging the edge resizes its lower radius and scales a custom upper radius to preserve the pillar profile. Edits snap to the active Pillar grid, preserve the existing base Y, reject undersized radii, and commit through undo/redo.
- Roof drawing snaps to the dock's Roof grid and preserves the configured base Y height as the roof eave plane. Built-in styles are `Flat`, `Shed`, `Gable`, `Hip`, and `Dome`. Flat exposes Rectangle and Polygon creation; Rectangle uses the two-corner/drag gesture, while Polygon captures vertices until the first is clicked again or Enter is pressed. Both Flat creation forms then share one editing model: drag any vertex, drag an edge and its two endpoints, Shift-click an edge to insert a vertex, Option/Alt-click a vertex to remove it, or drag the body to translate the footprint. A rectangular Flat roof promotes to polygon storage on its first vertex/edge shape edit, and undo restores rectangle storage. Floor and unclipped Flat polygon solids share the same polygon-prism generation path. Flat overhang creates one bounded miter point for each authored point, preserving outline topology even at acute corners; its closed top, underside, fascia, and collision follow that contour. Roof-cover clipping may add transient intersection points to generated mesh surfaces but never to authored/editor vertices. Shed, Gable, Hip, and Dome remain rectangular and keep rotated edge/corner editing. Shed, gable, hip, and dome use the configured face angle and derive generated height from roof run; Flat ignores angle. Dome uses a fixed low-poly elliptical cap with 16 radial facets and five vertical bands. Hip uses a horizontal ridge along the longer axis, with Gable Drop producing half-hip/jerkinhead variants while preserving face angle. Roof thickness extends downward and overhang expands beyond the authored footprint. Rotation and `R`/`Shift+R` apply to rectangle and pitched roofs; polygon Flat orientation comes from its vertices. Same-eave roofs remain separate authored nodes while height-aware polygon clipping removes covered render regions, closes cut edges with fascia, and supplies transient underside surfaces that clip intersecting walls. Edits snap to the active Roof grid, preserve eave-plane Y, reject invalid or zero-area footprints, refresh sibling roof and wall clipping, and commit through undo/redo.
- New wall spans merge into an existing collinear wall of matching thickness and height when their ranges overlap.
- Matching collinear wall spans from separate room nodes always use scene-order geometry ownership. The earlier wall keeps the shared wall mesh/collision, while duplicate overlap is clipped from the later room without changing either room's authored spans, even when generic intersection clipping is disabled.
- Door and window placement, movement, and resizing are allowed along matching collinear shared room boundaries on either wall, regardless of scene order: the wall that renders the shared span cuts the opening whether it is authored there or propagated from the clipped sibling. Crossings, T-junctions, and different-thickness overlaps remain blocked because their solid mass would fill the hole.
- An opening authored on one wall of a matching collinear overlap is propagated onto every collinear sibling that shares the span, in both scene-order directions. The earlier wall renders the shared mesh, so it cuts the visible hole; collision is generated on each collinear wall independently (it is not sibling-clipped), so each wall also cuts the opening from its own collision and the doorway is passable no matter which wall owns the opening node. The propagation maps the opening into each receiving wall's segment-local span as transient rebuild data and never edits authored spans or reparents the opening node; moving or resizing the opening refreshes the affected walls.
- Non-collinear walls that intersect (crossings, T-junctions, corners) remain separate `Wall3D` instances. `Building3D` always refreshes geometry-only sibling clip segments after wall creation, edits, undo, redo, and scene load. Each wall rebuilds only its own authored spans while using sibling spans on the same base plane as non-rendered cutters, removing buried side and top-cap geometry without reparenting children or deleting the other wall node. Same-height top-cap overlap resolves by scene order so only one wall owns the shared area; wall underside caps are not generated.
- Walls intersecting visible roof render regions preserve the authored wall volume below the roof's bottom surface and remove only the excess above it. Retained exterior faces keep horizontal outward wall normals with matching winding, and their contours split at eaves, slopes, cell-height crossings, and gable ridges. The covered clipping boundary emits no wall upper-face triangles because the roof already supplies that hidden surface. Roof-clipped collision uses the exact retained faces instead of flattening a whole span to its lowest roof sample. The clip follows flat, shed, gable, hip, dome, rotated, overhanging, and partially covered roof geometry; it is transient rebuild data and does not edit wall endpoints or extra segments.
- Authored wall segments, openings, and roofs emit clipping-relevant geometry changes. `Building3D` connects those signals and coalesces them into deferred transient wall/roof clipping refreshes, including changes made outside the plugin commit path.
- Saved wall, floor, stairs, rail, pillar, and roof meshes carry hidden authored-input and clipping-input signatures through `BuildingMesh3D`. Scene loading reuses matching meshes and rebuilds only generated collision/debug children; repeated clipping refreshes leave wall and roof meshes untouched until authored inputs or resolved clipping conditions change. Window and door openings cache their generated child-part mesh resources with an equivalent style-property signature. Scenes saved before these signatures existed rebuild once to establish the cache.
- Wall and roof editor commands refresh roof overlap followed by wall sibling/underside clipping in the same command, so committed, canceled, undo, and redo states never retain a partial cross-family clipping result while waiting for deferred signal handling.
- Window/prop placement and viewport picking are segment-aware: raycasts test every segment of a wall and previews align to the hit segment's frame.
- `BuildingOpening3D` children create rectangular wall holes without boolean operations. The wall compiles a split box-grid mesh around all openings. Shared opening dimensions, frame casing, and collision settings live on this base class.
- Window placement can create single windows, double windows, grid (colonial) windows, louvered windows, transom windows, arched-look windows, or frame-only openings. Window variants keep the full frame and add one or two translucent generated pane meshes inside the opening. Grid windows overlay frame-colored muntin bars dividing each pane into a grid of lights; louvered windows replace the flat pane with tilted horizontal slats; transom windows add a horizontal rail splitting an upper light off the top of the pane; arched windows add stepped frame-colored corner fillers that fake a low-poly arch on top of the pane.
- Door placement uses the same segment-aware opening workflow as windows, but cuts openings down to the wall base and can place single doors, double doors, glazed doors, cross glazed doors, panel doors, Dutch doors, single door frames, or double door frames. Door frames omit the bottom frame piece; door variants add one or two generated panel meshes inside the opening. Glazed doors split each panel into a solid lower section and a translucent upper glass lite separated by a rail; cross glazed doors add muntin bars across that glass; panel doors raise a recessed-panel inset grid on the solid face; Dutch doors split each panel into two stacked leaves with a mid rail.
- Styled-block details live on the narrowest relevant type. Family bases keep universal behavior and contain no style geometry branches; optional internal layers own properties shared by a real subset; and concrete pillar, roof, window, and door classes own style identity and style controls. Per-roof geometry strategies isolate reusable clipping/topology math by style. The dock instantiates a concrete class for every listed style instead of storing style identity as flags on a generic block. The normative future-block rules live in [`contract.md`](contract.md#building-block-style-pattern).
- The frame casing side is configurable per opening through the dock's Frame Sides control (window and door sections, persisted with the dock state) and the `BuildingOpening3D.frame_sides` property. `FRONT` (default) keeps the legacy single-sided casing on the placement face; `BOTH` centers the casing in the wall so trim shows and protrudes on both faces. `BOTH` mode uses `wall_thickness` (set from the wall the opening is placed on, defaulting from `frame_depth` for hand-authored openings) and `frame_protrusion` (how far the casing sticks out past each face). The opening rectangle the wall compiler cuts is unchanged by the casing side.
- Window and door openings stay axis-aligned to their wall face; opening rotation is not supported, so the frame visual always matches the cut rectangle.
- Window openings keep their segment-local position and face alignment when wall endpoints, joints, extra segments, or whole walls are dragged.
- Window placement snaps horizontally to the coordinator's grid step, and the opening's bottom edge sits at the configurable sill height from the dock (persisted with the dock state, default 0.9) instead of following the cursor's vertical position. The selected window style and dimensions persist with the dock state.
- In Window or Door tool mode, hovering over a placed opening highlights it blue (center) or yellow (edge). Clicking and dragging the center repositions it; dragging an edge resizes it — left/right edges adjust width symmetrically, top/bottom edges adjust height (Y locks to the opening's sill + half-height). Both snap to grid. Releasing commits via undo/redo; Escape or right-click cancels.
- In Wall tool mode, hovering near any segment endpoint highlights it yellow (resize); hovering a shared joint endpoint also shows an orange joint marker; hovering the middle highlights blue (move). Shift-clicking the middle of a wall span inserts a joint by splitting that `segments` entry into two connected resources. Dragging an isolated endpoint moves only the picked segment endpoint (grid-snapped); dragging a shared joint endpoint moves every connected segment endpoint at that joint; Option/Alt-dragging a shared joint endpoint detaches only the picked endpoint; dragging any single endpoint near another endpoint or joint snaps it there to connect. Dragging the middle translates the wall's authored segment set. If an endpoint drag collapses a span to zero length, release deletes that segment, promoting another entry to segment zero when needed; collapsing the only span deletes the wall node. If a drag edit crosses another wall, sibling clip geometry refreshes without merging wall instances; same-wall crossings are still normalized so the crossing point becomes an editable endpoint on both involved segments.
- For a four-span rectangular room, dragging the middle of one side resizes the room only along that side's perpendicular grid axis. Both adjacent corners follow the dragged side, the opposite wall remains fixed, and a zero-width or zero-depth room is rejected. Option/Alt-dragging the body of a rectangular room instead translates the whole room without resizing it, grid-snapped and committed through undo/redo. Open or irregular wall networks retain whole-body translation.
- Wall meshes duplicate vertices per face, carry vertex colors, use rough flat materials for hard low-poly face breaks, omit hidden underside triangles on the wall base plane, retain potentially visible down-facing header surfaces above openings, and generate opening-aware solid box collision cells so `HumanBody3D` is blocked by both sides of the wall instead of sliding through a thin mesh shell.
- Floor meshes duplicate vertices per face, carry vertex colors, use rough flat materials, split their top and bottom faces around stored holes, generate vertical inner faces for each hole, and generate concave mesh collision from the slab geometry.
- Stairs meshes duplicate vertices per face, carry vertex colors, use rough flat materials, expose real horizontal treads and vertical risers, omit the broad hidden underside quad, generate concave mesh collision from the remaining stepped faces, and add left/right side-wall box collision blockers inside the footprint edge strips under the treads. The front/back boundaries and side profiles still reach the configured underside depth. The side blockers are tagged with `stairs_side_wall_collision`, split per tread, and follow the stair profile, so characters cannot enter from the stair sides while the center lane remains climbable and characters already on a step can leave or jump off sideways without collision extending beyond the stair footprint. When enabled, a side's optional handrail is appended into this same generated mesh and concave collision as a straight raked run from the lower entry to the top tread, rather than stepping with each tread.
- Pillar meshes duplicate vertices per face, carry vertex colors, use rough flat materials, and generate cylinder collision from the outermost body or rim radius and height.
- The global Debug Display applies the same transient wireframe to wall, floor, stairs, rail, pillar, roof, multipart opening, preview, and marked prop meshes. It assigns a native spatial-shader `wireframe` material overlay directly to each source's RenderingServer instance, bypassing authored properties and avoiding edited-scene child mutations, CPU triangle extraction, and duplicate mesh allocation. Normal display retains back-face culling and depth testing so invisible surface triangles stay hidden; the explicit X-ray variant disables only depth testing. Color, visibility, and X-ray changes replace only temporary server-side overlay materials, preserve and restore any authored material overlay, and never rebuild source geometry, collision, or scene hierarchy.
- Wall mesh normals point outward, triangle winding follows Godot's `BoxMesh` convention, and wall materials use backface culling so lighting follows the generated face normals.
- Generated collision children are editor/runtime rebuild artifacts and should not be edited by hand.

## Edge Cases

- A wall shorter than half the active grid step, with an absolute floor of 0.1 units, is ignored.
- A room whose width or depth is shorter than half the active wall grid step, with an absolute floor of 0.1 units, is ignored.
- A rectangle floor whose width or depth is shorter than half the active grid step, with an absolute floor of 0.1 units, is ignored. A polygon floor requires at least three distinct vertices and successful simple-polygon triangulation.
- Floor holes must be fully inside one existing floor and cannot touch the outer slab edge. A hole may overlap or share an edge with another hole, including across the rect and polygon representations; overlapping or touching holes are merged into a single opening at mesh time the same way overlapping rect holes are unioned. Rectangle and polygon holes generate matching top/bottom cuts, vertical inner faces, and concave collision. When holes meet, overlap, or share an edge, their outlines are merged before the inner walls are generated, so no stray vertical wall is left where they join; a ring of holes that encloses a solid floor island keeps the island and its surrounding walls.
- A stairs footprint whose width or depth is shorter than half the active grid step, with an absolute floor of 0.1 units, is ignored.
- A pillar radius shorter than the active pillar minimum, with an absolute floor of 0.05 units, is ignored or restored.
- A rectangle roof whose width or depth is shorter than half the active grid step, with an absolute floor of 0.1 units, is ignored. A polygon Flat roof requires at least three distinct vertices and successful simple-polygon triangulation.
- Roofs remove covered render geometry only where their same-eave render areas overlap and the local roof surface is under another roof surface; style, roof angle, thickness, overhang, rotation, and color differences do not prevent height-aware clipping. Matching overhangs participate in render-area clipping even when committed footprints only touch or are separated by less than the combined overhang. Cover regions are stored as simplified face-intersection polygons when available, with rectangle bounds kept only for compatibility and quick visibility checks.
- Roof-to-wall clipping is based on visible roof render regions; covered roof regions do not clip walls after roof-cover refresh, and moving a roof away clears stale wall clip surfaces.
- Dragging an existing segment to exactly zero length deletes it through undo/redo; nonzero spans below the minimum length are still rejected and restored.
- Window and door openings are rejected when they leave the wall bounds, overlap another opening, or straddle another authored segment or sibling intersection clip (the crossing wall's solid mass would block the hole). Doors are allowed to touch the wall base; windows keep bottom clearance from the base.
- Child openings are assigned to the segment whose face shell they sit on (distance to the face, not the centerline), so openings near junctions stay on the wall they were placed against; the window tool also pins the hit segment index as metadata, with geometric assignment as fallback.
- The opening preview re-parents to whichever wall is hovered, so moving between separate walls in window or door mode places against the correct node.
- Prop placement can fall back to the ground plane when no wall target exists.
- The first wall click or placement commit can create a `Building3D` coordinator if the scene has none. Hover previews never mutate the scene or undo history.
- The first solid floor click can create a `Building3D` coordinator if the scene has none. Hole mode requires an existing floor target. Floor previews never mutate the scene or undo history.
- The first stairs click can create a `Building3D` coordinator if the scene has none. Stairs previews never mutate the scene or undo history.
- The first pillar click can create a `Building3D` coordinator if the scene has none. Pillar hover previews never mutate the scene or undo history.
- The first roof click can create a `Building3D` coordinator if the scene has none. Roof previews never mutate the scene or undo history.
- Editing an existing floor uses editor-time ray math against footprint bounds plus exact plan-polygon containment, skips stored hole interiors for floor picking, and does not access physics `direct_space_state` during GUI input.
- Editing existing stairs uses editor-time ray math against stair bounds and does not access physics `direct_space_state` during GUI input.
- Editing an existing pillar uses editor-time ray math against finite cylinders and does not access physics `direct_space_state` during GUI input.
- Editing an existing roof uses editor-time ray math against roof bounds plus exact Flat polygon containment and does not access physics `direct_space_state` during GUI input.
- Preview walls are tagged with preview metadata and never participate in intersection clipping.
- Wall spans whose base heights differ by more than 0.01 units are not merged; they stay separate nodes.
- Undoing or redoing wall creation, edits, and deletion refreshes sibling clip geometry so separate intersecting wall nodes keep their own children and authored spans.
- Overlapping collinear spans of matching thickness and height extend an existing segment instead of stacking a duplicate span.
- If the configured folder is missing, the prop palette falls back to `res://assets` when available, then `res://`.

## Architecture / Ownership

- The editor plugin lives under `addons/low_poly_building_editor/`.
- The plugin owns dock UI, the 3D viewport toolbar tool selection (added to `CONTAINER_SPATIAL_EDITOR_MENU`), viewport input forwarding, previews, and undo/redo packing.
- The dock owns the canonical tool-mode selection state; the viewport toolbar and editor selection changes drive it through the dock's `select_tool_mode()` entry point, and the plugin's `_on_tool_mode_changed` handler reflects the active mode back onto the toolbar buttons.
- Styled-tool dock sections keep common creation defaults visible while conditionally showing style-owned controls: side count for round/tapered pillars; angle and hip gable drop for applicable roofs; pane, grid, louver, transom, and arch controls for matching window styles; and leaf, glazing, grid, and inset controls for matching door styles. These values persist in editor project metadata and are applied to both previews and committed nodes.
- The plugin keeps building tools mutually exclusive with the native viewport modes by locating each named native Transform/Move/Rotate/Scale/Select button independently (`Transform`/`ToolTriangle`, `ToolMove`, `ToolRotate`, `ToolScale`, `ToolSelect`) from its editor icon first, then from the toggle button's tooltip text, then from the native shortcut when both fail (Transform Q, Move W, Rotate E, Scale R, Select V): it clears their highlight while a building tool is active and listens to their toggled, mouse, and button signals to deactivate the building tool when a native mode is chosen. If the native buttons cannot be located, the toolbar still works on its own without enforcing the exclusivity.
- A lightweight 3D viewport overlay plus root-level editor input capture handle placement clicks while a building tool is active so Godot's default select/transform mouse handling does not compete with wall, prop, or window placement. Tool-mode changes push active state into each overlay, and parent child-order notifications keep active overlays in front without frame polling. The root-level capture only acts over a 3D viewport that is currently visible on screen, so clicks in docks, the bottom panel, or other main screens (and hidden split-view viewports) are never intercepted.
- Viewport picking uses editor-time ray math against wall boxes, floor bounds and footprint polygons, stair bounds, finite pillar cylinders, and roof bounds plus a ground-plane fallback, avoiding `direct_space_state` access during editor GUI input.
- `Building3D` is the scene-owned root for one assembly. Multiple roots can coexist in one scene; Add Building creates and selects a new root, selecting a root or authored descendant changes the active tool target, and each root independently discovers its authored children, subscribes to their geometry-change signals, coalesces refresh requests, and delegates transient clipping calculations.
- `BuildingFactory` owns placement snapping, eight-way wall constraints, unique naming, concrete pillar/roof/opening style instantiation, opening property/transform configuration, and construction of wall, room, rectangle/polygon floor, stairs, standard rail, pillar, rectangle/polygon Flat roof, other roof styles, door, and window nodes from explicit values.
- `BuildingSpec` owns the typed, versioned generator input. `BuildingSpecCompiler` owns shared JSON loading, deterministic resolution, assembly, whole-building validation, authored ownership, and packing support. `generate_building.gd` owns single-scene command-line parsing and report I/O. `BuildingThumbnailRenderer` owns the reusable preview world, rendering, PNG validation, and contact-sheet composition; `generate_variants.gd` owns batch options and the variant manifest.
- `BuildingMesh3D` owns serialized generated-mesh cache signatures, rebuild counters used by validation, cached vertex/index extraction, and the transient wireframe entry point shared by wall, floor, stairs, rail, pillar, and roof nodes. `BuildingWireframe3D` owns recursive mesh discovery, temporary material-overlay state and restoration, and native wireframe shader materials; `BuildingOpening3D` and marked props reuse it for multipart geometry.
- The editor dock owns all new-element defaults and per-tool grids; selecting or editing tools does not mutate serialized `Building3D` properties.
- `WallGeometryResolver` transiently resolves collinear merge targets, intersecting walls, opening conflicts and propagation, and sibling wall clip inputs for one building root.
- `RoofGeometryResolver` transiently resolves roof overlap/visibility polygons, covered regions, and roof underside clip surfaces for one building root.
- `Wall3D` owns its zero-or-more authored `segments`, transient sibling and roof clip inputs, opening-to-segment assignment, and the mesh/solid collision rebuild.
- `Floor3D` owns rectangle and simple-polygon slab meshes, editable polygon holes, legacy rectangle holes, vertex colors, material, and generated collision.
- `Stairs3D` owns its stepped mesh, vertex colors, material, rotation, step count, generated collision, optional left/right side rails, and independently configured lower/upper tread-or-floor newels, which it appends into the same mesh/collision via the shared rail geometry strategy.
- `Rail3D` owns standard post-and-bar geometry, span transforms, material, cache signature, and generated collision, delegating post/bar mesh assembly to the shared `standard_rail_geometry_3d.gd` strategy so `Stairs3D` can reuse the identical geometry for a sloped run instead of a level span.
- `Pillar3D` owns universal pillar mesh/collision infrastructure; internal variable-sided and concrete style classes own subset/style properties and geometry decisions.
- `Roof3D` owns universal roof mesh/collision and clipping infrastructure; `FlatRoof3D` owns optional polygon footprint storage, while the sloped intermediate, concrete roof classes, and one-to-one style geometry strategies own subset/style properties and topology.
- `MergedWallMeshBuilder` (`merged_wall_mesh_builder.gd`) owns the plan-space clipping math that can render a subset of wall segments while using overlapping supplied same-plane segments as cutters, and the roof underside clipping that removes generated wall faces above visible roof bottoms without generating covered upper faces while preserving complete wall spans outside partial roof overlaps.
- `StandardRailGeometry` (`standard_rail_geometry_3d.gd`) owns the shared stateless post-and-bar rail assembly: default post-count/spacing distribution, tread-mid post position and base-height calculation, optional parallel per-post position/base/thickness/top-height layouts, piecewise horizontal/raked handrail spans, box/quad mesh appends, and embedding a canonical level rail profile into a caller-supplied run/up/side axis frame with an optional linear height rise. Posts remain upright with flat bases; regular post tops follow the raked handrail underside while newel tops remain flat beneath horizontal end sections. Every handrail underside segment is partitioned around its post footprints so both parts terminate on the same welded boundary with no overlapping internal face or wedge gap. The enclosed post top cap and hidden bottom cap are omitted while side faces retain welded top and base-contact edges. `Rail3D` calls the strategy with zero rise and no explicit post layout, so posts fall back to even spacing and uniform thickness on one level handrail. `Stairs3D` supplies a full layout based on tread centers/heights plus enabled tread- or floor-mounted newels and the horizontal-to-raked transition positions.
- `WallSegment3D` (`wall_segment_3d.gd`) is the typed resource for one wall span, including static helpers for collinear segment merging and intersection splitting.
- `Wall3D` owns generated mesh, vertex colors, opening-aware solid collision cells, and opening-driven rebuilds.
- `BuildingOpening3D` owns dimensions consumed by wall mesh generation, frame casing (`frame_sides`/`frame_protrusion`/`wall_thickness`), cached child-part mesh descriptors, low-level box/glass helpers, and collision assembly. `Window3D` and `Door3D` own universal category behavior; internal pane/leaf layers own reusable plain-part primitives; concrete styles own their exported tuning and generated style geometry. With `generate_collision` (default on) each primary solid part (frame jambs, door panels/leaves, glass panes, louver slats) parents a `StaticBody3D`/box `CollisionShape3D` so a closed door/window and the frame block the character; decorative overlays skip collision since the part behind them already carries it, and a frame-only opening stays passable because only the edge frame carries collision.

## Signals / Nodes / Data Flow

- Dock signals update active tool mode, global debug display, and wall/floor/stairs/rail/pillar/roof/prop/window settings, including the Wall tool's Wall/Room type, the Floor tool's Solid/Hole type plus Rectangle/Polygon style, and Flat Roof's Rectangle/Polygon footprint creation choice; the 3D viewport toolbar buttons and building-node editor selections call the dock's `select_tool_mode()` so the same `tool_mode_changed` signal carries selections from any editor entry point.
- The plugin forwards 3D viewport mouse/key input while a building tool is active.
- The plugin commits scene mutations through `EditorUndoRedoManager`.
- `Wall3D` subscribes to its segment resources and direct child openings, rebuilding and notifying `Building3D` when clipping-relevant authored geometry changes.
- In editor mode, `Wall3D` coalesces authored changes through one-shot rebuild requests: visual geometry updates on the next deferred turn while collision waits for the latest request's short timeout. No wall idle process loop is required.
- `Wall3D` captures direct child opening anchors before wall geometry edits and restores them into the updated segment frame so windows follow edited walls.
- Wall raycast hits carry a `segment` index so window and prop previews can target the correct authored span.

## Contracts / Boundaries

- The plugin is an editor authoring helper. It must not be wired into `main.tscn` or runtime gameplay flow.
- Scene-authored building content should remain normal Godot nodes under a `Building3D` coordinator.
- If wall segment storage, floor style/vertex/corner/hole storage, stairs footprint storage, pillar base/radius storage, roof footprint storage, opening child semantics, or intersection clipping behavior changes, update this file plus `contract.md`.

## Out Of Scope

- True 3D CSG boolean unions; intersection baking clips in plan space and assumes walls in a group share a base plane.
- Rich asset thumbnails beyond the basic scene palette list.
- Runtime gameplay interaction with authored low-poly buildings.
- Multi-storey generation, inferred internal floor plans, compound wings, prop selection, overlapping roof plans, and model-driven visual ranking are outside generator version 1.
